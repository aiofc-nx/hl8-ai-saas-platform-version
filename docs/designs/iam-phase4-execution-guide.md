# IAM Phase 4 执行计划（User Story 2 – 子域负责人规划交付）

## 目标概述

Phase 4 聚焦将规范扩展为子域负责人可直接使用的规划工具：完善里程碑计划表、职责矩阵与子域交付清单，使各团队能够依据规范拆解阶段任务并制定具体计划。

## 任务拆解

### Task T009：完善“里程碑计划”表

- **目标**：在 `docs/designs/iam-specification.md` 明确四个阶段的目标、交付物、质量门槛和建议时长。
- **行动项**：
  1. 对照 `docs/designs/saas-platform-iam-plan.md`，确保四阶段内容与计划文档一致。
  2. 补充每阶段的关键验证方式（如测试覆盖率、能力缓存指标、审计审查）。
  3. 若存在新的风险或依赖，注明在质量门槛或备注中。
- **完成标准**：表格信息完整，字段齐全；子域负责人可据此安排迭代周期和资源。

### Task T010：编制“职责矩阵”章节

- **目标**：在同一文档中形成每个角色的 RACI 表，为跨团队协作提供依据。
- **行动项**：
  1. 根据 Phase 2 的治理流程和已有职责总结（架构、子域、治理、平台基础设施、质量）确认角色列表。
  2. 标注 R（Responsible）、A（Accountable）、C（Consulted）、I（Informed），确保角色覆盖所有关键任务。
  3. 若新增角色（如安全团队、DevOps）对后续阶段有影响，可在备注中说明。
- **完成标准**：职责矩阵与实际团队结构对齐；评审时可直接引用判定责任归属。

### Task T011：更新子域交付清单

- **目标**：在 `specs/001-define-iam-spec/data-model.md` 的“子域交付清单”中补充依赖关系与质量门槛字段。
- **行动项**：
  1. 为租户、组织、认证、授权、共享内核、基础设施、接口层等子域列出关键交付物和依赖。
  2. 指明每个任务的质量门槛，如覆盖率、审计要求、性能指标、合规证明。
  3. 若子域之间存在强依赖，明确依赖来源（如认证依赖租户上下文）。
- **完成标准**：子域交付表内容可直接用于迭代计划，职责团队一目了然。

## 完成判定

- `docs/designs/iam-specification.md` 的“里程碑计划”与“职责矩阵”章已填充完毕，无空白列。
- `specs/001-define-iam-spec/data-model.md` 中的子域交付清单包含交付物、依赖、质量门槛等字段。
- 架构负责人或治理团队审阅后，确认子域负责人可据此制定阶段计划，并在变更日志记录 Phase 4 完成情况。
